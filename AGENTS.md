# Universal Change Tracking Workflow

> **Железное правило**: прежде чем что-то менять — прочитай лог предыдущих изменений.
> После каждого изменения — запиши что сделал. Нет записи = изменение не завершено.

---

## Шаг 0: Прочитай историю изменений (ПЕРЕД любым кодом)

### 0.1 Прочитай PROGRESS.md

```
Read("docs/PROGRESS.md")
```

Если задача уже помечена как выполненная — скажи об этом и остановись. Не переделывай.

### 0.2 Прочитай ВСЕ логи сессий

```
Glob("cursor/changelog-*.md")
Glob("cursor/bugfix-log-*.md")
```

Прочитай КАЖДЫЙ найденный файл (оба паттерна). Там записано:
- Какие файлы менялись ранее и почему
- Какие подходы не сработали
- Какие известные проблемы есть

**Не повторяй подходы, которые уже не сработали.**

### 0.3 Создай лог-файл текущей сессии

Если его ещё нет:

1. `Glob("cursor/changelog-*.md")` + `Glob("cursor/bugfix-log-*.md")` — определи последний номер среди ВСЕХ логов
2. Создай следующий: `cursor/changelog-NNN.md`

Шаблон:

```
# Changelog — Сессия NNN
**Дата**: YYYY-MM-DD
**Задача**: краткое описание задачи от пользователя
```

### 0.4 Подтверди основную папку и ветку

Перед любыми правками, deploy, commit/push или ответом "делаю" проверь:

```
pwd
git status --short --branch
git worktree list
```

Правильное рабочее место для изменений, которые должны быть видны в Xcode:

```
/Users/test/Downloads/AI Games for Roblox NEW Gold
```

Ветка должна быть:

```
main
```

Если пользователь спрашивает: **"ты работаешь в основной папке или worktree?"** — ответ должен быть:

```
Работаю в основной папке /Users/test/Downloads/AI Games for Roblox NEW Gold на ветке main.
```

Если текущий путь находится внутри `.claude/worktrees/...`, или в интерфейсе видно `main ← claude/...` / `Create PR`, это отдельный worktree/ветка. В таком случае не меняй файлы молча: сначала скажи пользователю, что Xcode в основной папке не увидит эти изменения, пока они не попадут в `main`, и получи явное разрешение работать в этом worktree или перенести изменения в основную папку.

### 0.5 Упрощённый режим Git/GitHub для этого проекта

Пока пользователь явно не попросит другой workflow, работай по простой модели:

1. **Рабочее место одно**: `/Users/test/Downloads/AI Games for Roblox NEW Gold` на ветке `main`.
2. **Xcode всегда открывает только эту основную папку**. Если правка сделана в worktree, Xcode её не увидит.
3. **Не работай в `.claude/worktrees/...` без прямого разрешения пользователя**.
4. **Не используй PR / `Create PR` как основной путь работы**. PR допустим только если пользователь отдельно попросил pull request.
5. **Незакоммиченные изменения можно тестировать локально**: они уже есть на Mac и видны Xcode, если сделаны в основной папке.
6. **Commit делай только после успешного теста/build/check** и только для файлов текущей задачи.
7. **Push в GitHub делай только после отдельной команды пользователя** вроде "залей на GitHub" / "push".
8. **GitHub здесь — удалённый сейв/checkpoint**, а не место для каждой промежуточной попытки.

Если есть сомнение, что сейчас открыт worktree или отдельная ветка, сначала остановись и объясни пользователю разницу простыми словами.

### 0.6 Firebase deploy идёт ПОСЛЕ commit'а

`firebase deploy --only functions:api` **не заменяет** `git commit`. Любой deploy должен быть в паре с уже сделанным commit'ом тех же файлов.

**Канонический способ deploy в этом проекте — wrapper-скрипт**, который сам проверяет §0.6 + §0.7 на уровне shell:

```bash
bash scripts/safe-deploy-functions.sh
```

Скрипт блокирует deploy если:
- Working tree содержит uncommitted правки в `apps/functions/src/` или `packages/shared/`.
- Параллельный deploy финишировал меньше 3 минут назад (lock-файл `.firebase-deploy.lock`).

Если есть основание обойти (емерженси): `FORCE_DEPLOY=1 bash scripts/safe-deploy-functions.sh` — но это записывается в changelog с обоснованием.

**Если вызываешь `firebase deploy` напрямую (без wrapper'а)** — обязан соблюдать тот же порядок вручную:

1. Правка файлов в `apps/functions/src/*` (или связанных пакетах).
2. `npm run build --workspace apps/functions` ✅.
3. Запись в `cursor/changelog-NNN.md`.
4. **`git commit` соответствующих файлов** (по правилу §0.5 п.6 — после успешной проверки).
5. **Только теперь** `firebase deploy --only functions:api`.
6. Health check `/api/health` + запись факта deploy в changelog.

**Почему это важно**: если deploy уходит без commit'а, prod-функции содержат правки, которых нет в git history. Следующая сессия Codex/Claude читает `git log`, не видит этих правок, и может пытаться чинить «снова с нуля» уже исправленное место — пользователь видит «по кругу одно и то же».

**Если commit ещё нельзя сделать** (проверка не прошла, build падает, в working tree есть посторонние правки не по этой задаче) — **deploy тоже нельзя**. Сначала разобраться с commit'ом, потом deploy.

**Push в origin** — отдельный шаг по правилу §0.5 п.7, deploy от него не зависит.

### 0.7 Дождись параллельных сессий: deploy и коммиты

В этом проекте часто работают **несколько агентов одновременно** (Claude и Codex в разных чатах). Если две сессии параллельно деплоят или коммитят, одна перезапишет работу другой → пользователь увидит «фиксы пропали» / «по кругу».

**Перед `firebase deploy`:**

1. Проверь, не идёт ли уже deploy в эту секунду:
   ```bash
   firebase functions:log --only=api -n 30 | grep -oE 'api-00[0-9]+-[a-z]+' | sort -u | tail -3
   ```
   Если ревизия создалась < 3 минут назад — параллельная сессия только что задеплоилась. **Подожди 2-3 минуты**, переcheck каждые 30-60с.
2. **Никогда не запускай deploy поверх активного deploy** — Cloud Run отвечает HTTP 409 `unable to queue the operation`.
3. Перед своим deploy сверь `src` ↔ `dist`. Если параллельная сессия откатила твои правки в `src` — сначала верни их, закоммить, потом деплой.
4. Если конфликт повторяется (3+ круга) — попроси пользователя приостановить параллельные сессии на этот проект.

**Перед `git commit`:**

1. `git status --short` — посмотри, нет ли в working tree модификаций, которые **сделал не ты** (другая сессия). Если есть незнакомые правки — не стажь их слепо, разберись чьи они (по содержимому или последним changelog'ам).
2. Если видишь файл `.git/index.lock` или ошибку `Another git process seems to be running` — **подожди 30-60с**, не убивай lock. Это идёт коммит другой сессии.
3. `git log --oneline -3` — проверь, не появились ли коммиты, которых ты не делал, с того момента как начал работу. Если да — твой `git log` уже не свежий, перечитай последние changelog'и и адаптируй план.
4. Стажь только файлы своей задачи по имени (`git add path/to/file`), не `git add .` — иначе утащишь чужие правки в свой коммит.

**Перед `xcodebuild` (или любой iOS-сборкой через CLI):**

1. **НЕ запускай `xcodebuild` против `apps/ios/AIGoldRoblox.xcodeproj`, если пользовательский Xcode уже открыт на том же проекте.** DerivedData один, и `build.db` имеет SQLite-lock — параллельный билд получит `unable to attach DB: database is locked`, **и Xcode пользователя тоже сломается на этой же ошибке**.
2. Проверка: `pgrep -fl "Xcode.app/Contents/MacOS/Xcode"` → если Xcode запущен, и он смотрит на этот проект (`osascript -e 'tell application "Xcode" to get path of active workspace document'`), **не запускай xcodebuild**.
3. Альтернатива: попроси пользователя закрыть Xcode временно, или используй **отдельный** DerivedData: `xcodebuild ... -derivedDataPath /tmp/aigold-claude-build`. Тогда твой билд не лочит пользовательский кэш.
4. После CLI-билда в `/tmp` — не забудь удалить временный DerivedData, чтобы не копить мусор.
5. Если уже запустил xcodebuild и видишь lock-conflict — кильни **свой** процесс (`kill <PID>`), не пользовательский Xcode.

**Перед самым первым действием в сессии** (preflight):

- `git status --short --branch` → понять, чьи правки уже в working tree.
- `ls -lat cursor/changelog-*.md | head -3` → если последний changelog моложе твоего старта на минуты — другая сессия активна, не лезь в её файлы без явного разрешения пользователя.
- `pgrep -fl xcodebuild` → если активный CLI-билд работает, не запускай свой; жди или используй `-derivedDataPath /tmp/...`.

Это правило в паре с §0.6 (commit перед deploy) убирает корень «по кругу»: параллельные сессии больше не перетирают чужие правки молча.

## Шаг 1: Анализ перед изменением (ОБЯЗАТЕЛЬНО)

Прежде чем редактировать любой файл:

1. **Прочитай файл целиком** (или ±50 строк вокруг целевого места для файлов >500 строк)
2. **Пойми контекст**: что делает этот код, от чего зависит, что зависит от него
3. **Определи scope**: какие файлы затронет изменение. Меняй ТОЛЬКО то, что относится к задаче
4. **Проверь логи**: не ломалось ли это место раньше? Не откатывается ли предыдущий фикс?

**ПРАВИЛО**: каждое "попутное улучшение" вне scope задачи = риск регрессии. Не трогай то, что не относится к текущей задаче.

## Шаг 1.5: Исследование лучших практик (ПЕРЕД написанием кода)

> **Не выдумывай решение из головы — сначала проверь, как это делают другие.**

Перед тем как писать или менять код, выполни исследование через `WebSearch`:

### Что искать

1. **Best practices** — лучшие практики для конкретной задачи
   - Пример: `"SwiftUI navigation best practices 2025"`, `"React form validation best practices"`
   - Ищи именно для текущего стека и версии (iOS 18, React 19, Python 3.12 и т.д.)

2. **Работающие open-source решения** — готовые библиотеки и примеры
   - Пример: `"best open source Swift network layer library github"`, `"top rated React date picker npm"`
   - Проверяй: звёзды на GitHub (>500), дата последнего коммита (<6 месяцев), количество issues
   - Если есть подходящая библиотека с хорошей поддержкой — предложи её вместо самописного решения

3. **Актуальная документация** — API меняются, методы депрекейтятся
   - Пример: `"StoreKit 2 purchase API 2025"`, `"Next.js 15 app router migration guide"`
   - Используй `WebFetch` для чтения официальной документации, если нашёл нужную страницу

4. **Известные баги и подводные камни** — чужие ошибки экономят твоё время
   - Пример: `"iOS 18 StoreKit purchase known issues"`, `"React 19 useEffect cleanup bug"`
   - Проверяй Stack Overflow, GitHub Issues, форумы разработчиков

5. **Сравнение подходов** — если есть несколько вариантов решения
   - Пример: `"Redux vs Zustand vs Jotai comparison 2025"`, `"CoreData vs SwiftData performance"`
   - Выбирай подход, который лучше подходит к уже существующим паттернам проекта

### Когда искать

- **Всегда** — при добавлении новой фичи или интеграции
- **Всегда** — при использовании API/библиотеки, в которой не уверен на 100%
- **Всегда** — при выборе между несколькими подходами к решению
- **Всегда** — при фиксе бага, который не решается с первой попытки
- **Можно пропустить** — при тривиальных правках (опечатка, переименование, удаление неиспользуемого кода)

### Как фиксировать найденное

Запиши результаты исследования в лог сессии:

```
### Исследование: [тема]
- **Запросы**: что искал
- **Лучшие практики**: краткие выводы
- **Рекомендуемые решения**: библиотеки/подходы с ссылками
- **Подводные камни**: на что обратить внимание
- **Выбранный подход**: что решил использовать и почему
```

**НЕ копируй код вслепую** — адаптируй найденное решение под паттерны текущего проекта.

## Шаг 2: Внесение изменений

### Принципы

- **Фикси причину, не симптом** — разберись в root cause
- **Минимальный diff** — меньше изменений = меньше рисков
- **Сохраняй паттерны проекта** — не переписывай стиль, не меняй архитектуру без явного запроса
- **Не удаляй код** без понимания зачем он был добавлен (проверь в логах)
- **Предпочитай проверенные решения** — если есть популярная библиотека/паттерн, не изобретай велосипед

## Шаг 3: Запись в лог (ПОСЛЕ КАЖДОГО изменения)

Сразу после внесения изменений добавь запись в `cursor/changelog-NNN.md`:

### Формат записи

```
## [N]: Краткое описание изменения
- **Файлы**: перечисли все изменённые файлы с указанием что именно менялось
- **Причина**: почему потребовалось это изменение
- **Что было**: краткое описание состояния ДО изменения
- **Что стало**: краткое описание состояния ПОСЛЕ изменения
- **Затронутые зависимости**: какие другие части кода могут быть затронуты
- **Результат**: выполнено / частично / требует доработки
```

### Пример записи

```
## 1: Исправлена валидация email в форме регистрации
- **Файлы**: src/utils/validation.ts (функция validateEmail, строки 45-62)
- **Причина**: regex не принимал домены с дефисом (user@my-domain.com)
- **Что было**: regex `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` — не учитывал сложные домены
- **Что стало**: regex обновлён, добавлен тест-кейс для доменов с дефисом
- **Затронутые зависимости**: RegisterForm.tsx, ProfileEditForm.tsx (используют validateEmail)
- **Результат**: выполнено
```

### ВСЕГДА записывай в лог

- Даже если изменение было в одну строку
- Даже если откатил изменение — запиши "откат" и почему
- Даже если подход не сработал — запиши "не помогло" чтобы не повторять
- Если задача требует внешнего действия (настройка сервера, ключи, доступы) — запиши как ВНЕШНИЙ БЛОКЕР

## Шаг 5: Стоп-проверка

Перед тем как сказать "готово":

1. **Перечитай изменённые файлы** — убедись что всё корректно
2. **Проверь линтер** — `ReadLints` на всех изменённых файлах
3. **Проверь лог** — все изменения записаны
4. **Проверь PROGRESS.md** — статус обновлён

## Шаг 6: Git commit / push только после успешной проверки

> GitHub должен получать только проверенные checkpoint-состояния, а не каждую промежуточную попытку.

### Правило commit

- **Не коммить после каждой мелкой правки.**
- Сначала внеси scoped-изменения, запиши их в `cursor/changelog-NNN.md`, обнови `docs/PROGRESS.md` при завершении задачи.
- Запусти релевантную проверку:
  - backend/functions: `npm run build:functions` или более узкий workspace build, если scope только там
  - iOS: `xcodebuild ... build` для изменений Swift/Xcode project
  - docs/config-only: минимум `git diff --check -- <изменённые файлы>`
- **Commit разрешён только после успешной проверки.**
- Если проверка падает — не коммить и не пушь; сначала исправь причину или явно сообщи пользователю, что состояние не готово.

### Правило push

- **Push в GitHub делай только после успешного commit.**
- Не делай push, если последний commit не относится к проверенной задаче или сборка/тесты не проходили.
- Stage только файлы текущей задачи. Не добавляй параллельные/чужие dirty-файлы в commit без явного запроса пользователя.

## Stop-loss механизм

**Если изменение не даёт результата после 3 попыток** — СТОП:

1. Запиши в лог ВСЕ три попытки
2. Проанализируй: может быть проблема не в том файле/месте?
3. Перечитай весь flow задействованного кода от начала до конца
4. Если после 5 попыток не решается — запиши `"ЭСКАЛАЦИЯ"` и сообщи пользователю
